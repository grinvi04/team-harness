# 개발자 워크플로 가이드 스펙

## 1. 목표 & Why

`intro.html`의 개념 소개와 세부 표준 사이에서, 소비 프로젝트 개발자가 지금 해야 할 작업과 다음 절차를
빠르게 찾을 수 있는 길잡이를 제공한다. **성공 기준: 개발자가 5분 안에 일반 개발·수정·머지·긴급 수정·릴리즈
경로와 막혔을 때 확인할 정본을 찾을 수 있다.**

## 2. Scope

- **In:** 일상 작업 흐름, 상황별 skill 선택, 브랜치 흐름, 가드 대응, 작업 완료 체크리스트.
- **Out:** 최초 설치 상세, 각 skill의 내부 구현, 정책 결정 근거, team-harness 유지보수 절차.

## 3. 기능 요구사항 + 수용기준

- **AC-1 (역할):** WHEN 개발자가 문서를 열면, the document SHALL 소개·설치·세부 정책 문서와 자신의
  역할 차이를 먼저 설명한다.
- **AC-2 (작업 경로):** WHEN 새 기능·기능 수정·PR 머지·hotfix·release 상황을 찾으면, the document SHALL
  시작 위치, 사용할 절차, 완료 결과를 한눈에 보여준다.
- **AC-3 (실행 예시):** WHEN 일반 기능 개발 흐름을 따르면, the document SHALL 요청부터 plan, 구현, PR,
  review gate, develop 반영까지의 관찰 가능한 단계를 순서대로 보여준다.
- **AC-4 (문제 대응):** IF 가드나 CI가 작업을 막으면 THEN the document SHALL 우회 대신 올바른 다음 행동과
  troubleshooting 정본을 안내한다.
- **AC-5 (드리프트 방지):** WHILE 세부 정책이 바뀌더라도, the document SHALL 요약만 유지하고 상세 계약은
  기존 정본 문서와 skill에 링크한다. 모든 로컬 링크는 실제 파일로 해석돼야 한다.

## 4. 제약 / 비기능

- Markdown 한 파일로 유지하고, 처음 읽는 개발자가 위에서 아래로 5분 안에 훑을 수 있는 분량으로 작성한다.
- Claude Code와 Codex의 UI 차이보다 동일한 결과 계약을 중심으로 설명한다.

## 5. 경계 / Do-Not

- ✅ 해도 됨: 대표 작업 예시와 판단표를 제공한다.
- ⚠️ 먼저 물어봐: 기존 정본 문서의 정책을 변경한다.
- 🚫 절대 금지: skill 절차를 복제해 별도의 정본을 만들거나 가드 우회법을 정상 경로처럼 안내한다.
