# AI 협업 가이드

이 하네스의 존재 이유 — 5–10명이 각자의 방식으로 AI를 쓰면 코드 편차가 AI 도입 전보다 커진다.
도구가 무엇이든(개발팀 표준 = Claude Code, 타 팀 Codex/Gemini 가능) 아래 규칙은 공통이다.

## 대원칙

> **AI가 만든 결과물의 책임은 그것을 PR에 올린 사람에게 있다.**

- AI는 작성자(author)이고 당신은 그 작성자를 검수하는 시니어다 — 검수 없는 머지는 본인 책임
- **이해하지 못한 코드는 머지하지 않는다** (설명 가능성 원칙): 리뷰에서 "왜 이렇게 했나"에
  답하지 못하면 그 코드는 아직 당신 것이 아니다
- AI 사용을 숨기지 않는다 — 커밋에 `Co-Authored-By: <도구>` 표기 유지

## 작업을 맡기는 방법

- **명세를 포함해 위임한다**: 목표 + 제약(이 표준 문서들) + 검증 기준(통과해야 할 테스트)을
  첫 지시에 담는다. "알아서 해줘"는 토큰 낭비와 재작업의 지름길
- **한 PR = 한 작업**: AI는 시키면 끝없이 만든다 — 요청 범위 밖 리팩터링·기능 추가는 거절하고 분리
- **병렬 에이전트는 worktree로 격리한다**: 여러 에이전트가 같은 체크아웃을 공유하면 브랜치 전환·reset이
  서로의 미커밋 작업을 clobber한다(erp 실측 — 초기 동시 작업 사고). 동시에 코드를 수정하는 에이전트는
  각자 git worktree(또는 격리된 작업공간)에서 돌리고, 머지는 순차로(뒤처진 브랜치는 base 최신화 후)
- 막연한 탐색보다 **TDD 루프에 태운다**: 실패하는 테스트를 먼저 (사람이 또는 AI가) 확정하고,
  통과를 목표로 작업시키면 결과 검증이 자동화된다
- 규약은 repo의 `AGENTS.md`가 단일 출처 — AI에게 별도 장문 지시 대신 AGENTS.md를 채워라
- **생성 문서는 repo에 커밋한다**: AI가 만든 계획·설계 문서(`/plan` 스펙, `/milestone` 추적,
  설계 결정 등)는 도구 로컬(예: `~/.claude/plans`)에 두지 말고 프로젝트 `docs/` 아래에 커밋한다.
  로컬 캐시는 노트북·도구·세션이 바뀌면 유실 — repo에 있어야 누가·어디서 이어받아도 일관된다.
  각 repo의 `AGENTS.md`는 자기 `docs/` 경로(스펙·마일스톤)를 명시한다.

## 금지 사항 (전 도구 공통)

- **시크릿·고객 개인정보·미공개 재무 데이터를 프롬프트에 투입 금지** — 외부 AI 서비스 전송은 유출이다.
  연동 테스트에 실제 키가 필요하면 환경변수 참조로 우회 (값을 채팅에 붙여넣지 않는다)
- 운영 DB·운영 인프라에 대한 조작을 AI에게 위임 금지 — 사람이 직접
- 가드 우회(`--no-verify`, 권한 강제 승인)는 긴급 상황만 + 팀 리드 보고 (`pre-commit` 훅 메시지 참조)
- AI가 생성한 마이그레이션·권한 코드·금액 계산 로직은 **사람 검토 필수 지정 영역**

## 거버넌스 (도구 설정)

- 개발팀(Claude Code): repo clone → `docs/onboarding.md` §B — harness-guard 플러그인이
  가드·게이트·커맨드를 일괄 제공한다. 플러그인 미설치 상태로 작업 금지
- 타 도구 팀: `AGENTS.md` 준수 + 계층 0(branch protection·CI)이 도구 무관하게 강제
- git-flow 작업은 수작업 대신 플러그인 커맨드 사용: `/feature-merge`, `/hotfix`, `/release`
  (게이트 절차 누락 방지 — 절차의 단일 출처는 `pr-review-gate` 스킬)
- **스킬 사용 순서**: 호출 전에 skill.md를 먼저 읽는다.
  `disable-model-invocation: true`가 있으면 Skill 도구로 호출하지 말고,
  skill.md의 bash 명령을 직접 실행한다 (모델 판단 개입 없이 절차 그대로 수행).

## AI 생성 코드의 리뷰

- 사람이 쓴 코드와 **동일한 기준** (`code-review.md`) — "AI가 짰으니 대충"도 "AI가 짰으니 의심"도 아님
- 단, AI 특유의 실패 모드를 중점 확인:
  - **과잉 구현**: 요청 안 한 추상화·옵션·방어 코드 → 삭제 요구
  - **그럴듯한 오답**: 존재하지 않는 API·옛 버전 문법 → 실행·테스트로만 검증 (눈으로 통과시키지 말 것)
  - **경계 위반**: 모듈 직접 참조, 계층 역류 → ArchUnit이 잡지만 리뷰에서도 확인
  - **테스트의 자기 충족**: 구현을 베낀 테스트(항상 통과) → 테스트가 명세를 검증하는지 확인
- PR 본문에 AI에게 준 핵심 지시를 요약 기재 (리뷰어가 의도를 알 수 있게)
