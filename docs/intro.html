왜 만들었나

AI 코딩 에이전트는 속도를 높이지만, 팀의 편차도 함께 증폭한다. 10명이 각자의 방식으로 Claude를 쓰면 코드 일관성은 도입 전보다 나빠진다.

또 하나의 문제: AI는 부탁을 받았을 때만 규칙을 따른다. 프롬프트에 "main에 직접 커밋하지 마"라고 써도, AI가 빠른 경로를 선택하면 순식간에 무너진다.

핵심 통찰 — 프롬프트 지침은 강제가 아니다. 우회할 수 없는 구조만이 강제다.
branch protection과 CI는 AI도, 인간도, 어떤 도구도 건너뛸 수 없다.

설계 원칙

1. 계층적 방어 (Defense in Depth)

강제력의 원천을 GitHub 서버까지 내려보낸다. 위 계층이 우회되어도 아래 계층이 막는다.

2. AI 중립성

계층 0–1은 Claude Code, Codex, Cursor 등 모든 도구에 동일하게 작용한다. harness-guard 플러그인(계층 2–3)도 Claude Code와 Codex에서 각 런타임에 맞는 경로로 작동한다.

3. 단일 출처

규약은 각 repo의 AGENTS.md 하나. 중복·복붙 없이 이 파일을 업데이트하면 AI가 따른다. 절차는 team-harness의 스킬 파일이 단일 출처 — 개별 repo에 복사하지 않는다.

4. 배포는 버전으로

플러그인 버전 업 하나로 모든 repo에 최신 스킬·가드·에이전트가 배포된다. 파일 복사·수동 동기화 없음.

스킬 16종

스킬은 요청과 현재 상태에 맞는 metadata trigger로 선택될 수 있다. 다만 스킬 선택은 새 권한을 만들지 않는다. 머지·릴리즈·운영 변경 같은 외부 상태 변경은 사용자가 맡긴 범위와 각 게이트를 그대로 따라야 한다.

개발 워크플로

/plan
수동 전용effort: high
기능 개발 전 스펙·태스크 분해. 의도·수용기준·롤백 가능한 태스크를 docs/specs/에 박제. git 없음 — 계획 전용.
/feature-add
수동 전용effort: high
TDD 기반 신규 기능 개발. Red → Green → Refactor. 브랜치 생성·테스트 계약·구현·커밋까지. 스택 무관.
/feature-modify
수동 전용effort: high
기존 기능 수정. 변경 대상만 RED, 나머지 GREEN 유지. 회귀 방지에 집중.
/pr-create
Claude 자동 가능
base 자동감지(develop 있으면 develop, 없으면 기본 브랜치) PR 생성 단일 프리미티브. 맨손 gh pr create 대체 — develop 없는 main 기반 repo도 한 경로로. feature-merge·hotfix·release가 이 스크립트를 호출.
/feature-merge
수동 전용effort: medium
feature/* · fix/* 브랜치를 develop에 머지. 품질 검증 → PR 생성 → pr-review-gate 절차 경유.

릴리즈 워크플로

/release-check
수동 전용effort: max
릴리즈 전 병렬 검증. Agent A (품질·lint·test·build) · Agent B (보안 리뷰) · Agent C (DB 마이그레이션·표준) 동시 spawn. 전부 ✅여야 /release 진행 가능.
/release
수동 전용effort: high
release-check 통과를 전제로 실행. release 브랜치 생성 → 버전 범프 → main PR → 태그 → develop back-merge PR → 배포 헬스체크.
/hotfix
수동 전용effort: high
운영 긴급 수정. main 기준 hotfix 브랜치 → PR → 태그 → develop back-merge PR. git flow를 건너뛰지 않는다.

가드 훅

모든 Bash 명령 실행 전 PreToolUse 훅이 guard.sh를 실행한다. 차단 시 Claude가 멈추고 사유를 출력하며, 차단 이력을 ~/.claude/hooks/guard-block.log(session_id·cwd·명령 — 크레덴셜·토큰은 마스킹)에 기록해 멀티세션 위반 시도를 감사할 수 있다.

⛔ main/develop 직접 커밋 금지
현재 브랜치가 main·develop이면 git commit 차단. feature/fix/hotfix/release 브랜치에서만 커밋 가능.
⛔ main/develop force push 금지
--force · -f · +main refspec 감지 시 차단. 히스토리 변경 필요하면 팀장에게 직접 요청.
⛔ git reset --hard 금지
미커밋 변경사항 전체 삭제 위험. 사용자가 직접 실행해야 함 — Claude가 대신 실행 불가.
⛔ 검증기·마이그레이션 삭제 금지
테스트 파일과 마이그레이션(db/migration/·alembic/versions/·prisma/migrations/) 삭제를 차단 — 게이트를 무력화하고 "빌드 성공"으로 위장하는 회귀를 막는다.
⛔ 프로젝트 루트·src·app rm -rf 금지
심링크 경로까지 정규화해서 체크. /tmp ↔ /private/tmp 같은 우회도 차단. node_modules 포함.
⛔ npm install -g 금지
글로벌 Node 환경 오염 방지. 로컬 설치(--save-dev) 또는 npx 사용.
⛔ 맨손 gh pr create·gh pr merge 금지
PR 생성·머지는 래퍼 스크립트(pr-create.sh·pr-merge.sh) 경유만 — 반사적 맨손 gh 차단. 스킬이 스크립트를 호출(내부 gh는 자식 프로세스라 훅에 안 걸림).
my-project — main 브랜치
$ git commit -m "급해서 그냥"
⛔ [guard] main/develop 직접 커밋 금지
   해결: feature 브랜치에서 작업 후 /feature-merge 사용

$ git push origin main --force
⛔ [guard] main/develop force push 금지
우회 시나리오 — --no-verify로 pre-commit을 건너뛰어도, cd other-repo && git commit으로 경로를 바꿔도, guard.sh는 마지막 cd 기준으로 repo를 재탐지해 차단한다. 마지막 보루는 계층 0(branch protection)이다.

모델 티어링

작업 성격에 따라 세 모델을 분리 사용. 비용을 낮추면서 판단이 필요한 곳에 더 강한 모델을 투입한다. 정책의 단일 출처는 docs/model-tiering.md.

Haiku
  • 파일 스캔·읽기
  • git diff 조회
  • 진행률 조회
  • 헬스체크 명령
  • (release Phase 5 헬스체크)
Sonnet
  • 기능 구현·테스트
  • 품질 검증 (Agent A)
  • 마이그레이션 안전 감사 (Agent C)
  • lint·test·build
  • (general-purpose 기본 위임)
Opus (security-reviewer)
  • 보안 리뷰 (Agent B)
  • XSS·SQLi 탐지
  • 시크릿 유출 패턴
  • 설계·다각도 검증
  • 오케스트레이션
  • (verifier·Plan·메인 세션 기본값)
타입이 곧 티어다 — 모델은 타입을 따라온다. 판단 없는 read-only 작업(조회·헬스체크)은 subagent_type: Explore(읽기전용 도구 = 최소권한, Haiku). 파일을 수정하거나 빌드하면 general-purpose(Sonnet) — 단순 집계라도 파일을 쓰면 Explore 불가. general-purpose에 haiku를 붙이는 건 타입↔티어 불일치라 지양한다.

왜 Agent C는 Sonnet인가 — DB 마이그레이션 안전 감사(out-of-order·forward-only·증분 적용 관점)는 릴리즈 게이트의 판단형 작업이라 Haiku로 내리면 회귀. 반면 Phase 5 배포 후 헬스체크는 명령 실행·상태 확인뿐이라 Explore(Haiku)가 맞다.

release-check의 Agent A·B·C는 동시에 spawn된다 — 병렬 실행으로 전체 검증 시간을 단축.

비용 인식 — 읽기·검색은 최저 티어(Explore/Haiku), 빌드·구현은 Sonnet에 위임한다. 병렬성 있는 무거운 다단계는 Workflow가 stage별 모델을 결정론적으로 지정하고, 계획과 실행을 분리할 땐 opusplan(계획=Opus·실행=Sonnet)을 쓴다. 다만 가장 큰 비용 축은 모델 선택이 아니라 세션 길이다 — 긴 단일 세션은 누적 컨텍스트를 매 턴 캐시-리드해 Opus 비용을 키운다(실측상 최대 축). 페이즈 전환 시 /clear, 컨텍스트가 커지면 /compact로 리셋한다. 위임·세션 위생의 단일 출처는 docs/model-tiering.md.

Git Flow

모든 변경은 PR로만 흐른다. 직접 커밋·push 경로는 없다 — 시도하면 3겹(guard→pre-commit→branch protection)에서 차단된다.

feature/*
fix/* /feature-merge → develop /release → main + 태그
main /hotfix → main + 태그 back-merge PR → develop
릴리즈·핫픽스의 develop back-merge도 PR로 진행된다. 긴급 상황이어도 direct push는 없다.

PR 머지 게이트

/pr-review-gate 스킬이 단일 출처. feature-merge · hotfix · release 모두 이 절차를 참조한다 — 복붙 없음, 드리프트 없음.

게이트 래퍼

AI 리뷰(/code-review 스킬 — 구독 포함·외부 봇/시크릿 불필요)가 변경분을 검토해 제안을 남긴다. 제안을 처리하고 스레드를 resolve한 뒤, 팀 모드(리뷰어 有)에선 사람 승인이 별도로 필요하다 — AI 리뷰가 사람을 대체하지 않는다. 솔로 표준(승인요건 0)에선 CI·스레드 resolve가 우회불가 게이트이고 이 단계는 생략된다(enforce_admins=on이 소유자·AI에게도 CI-green을 강제).

솔로→팀 전환은 set-branch-protection.sh --approvals N 한 줄 — main에만 리뷰 승인 N을 걸고(승인 후 push 시 stale 승인은 dismiss_stale_reviews로 무효), develop은 승인0 유지해 pr-merge.sh --auto 무프롬프트 자동머지를 보존한다. 신규 repo 생성 땐 걸지 않는다 — 소유자 1명이면 자기 PR 자기승인 불가로 첫 PR이 데드락되기 때문. 멤버 합류 후 올린다.

솔로 개발자 — GitHub에서 자기 PR을 승인할 수 없다. /solo-merge가 CI·스레드 확인 등 품질 게이트를 모두 통과시킨 뒤 승인 요건만 required_pull_request_reviews 일시 삭제로 우회하고 즉시 복구·검증한다(--admin/enforce_admins 토글은 GitHub 변경으로 더 이상 작동 안 함).

온보딩

팀원 (3분)

  1. clone — git clone <repo> && cd <repo>
  2. git hook 연결 — git config core.hooksPath .githooks
  3. Claude Code 실행 — claude · 첫 실행 시 harness-guard 플러그인 신뢰 확인 → 완료

신규 프로젝트 셋업 (리드, 1회)

  1. GitHub repo 생성 (public 권장 — Free plan branch protection 확보)
  2. ~/team-harness/scripts/new-repo.sh 실행 → 스택 선택 → AGENTS.md·CI·.claude/ 자동 생성
  3. branch protection 설정: docs/onboarding.md 체크리스트 참조
  4. 초기 커밋 push → develop 브랜치 생성 → branch protection develop에도 적용
  5. 팀(리뷰어 ≥1): 멤버 합류 후 set-branch-protection.sh <repo> --approvals 1로 main에 승인 요건 추가 (develop은 0 유지)

Codex 호환성

team-harness는 Claude Code 원본을 보존하면서 Codex에 전용 overlay·hook adapter·agent 설정을 적용해 동일한 거버넌스 의미를 제공한다.

✓ Codex 동작
AGENTS.md · CI · git hooks · stack rules · skill 16종 · custom agents · command hooks
⚠ 런타임 차이
Claude prompt hook의 핵심 guidance·known egress 차단을 Codex command hook·adapter로 매핑한다. 구현은 다르며 검증된 fixture의 보호 결과를 기준으로 한다.
✗ 미검증 도구
Cursor·Gemini 등은 plugin/hook parity를 검증하지 않았다. AGENTS.md·git hook·CI·branch protection 범위로 제한한다.
결론 — Codex는 v0.55.0 이상 플러그인, managed requirements, hook trust가 준비되면 Team Harness skill·agent·보안 가드를 사용한다. 최종 강제는 도구와 무관한 branch protection·CI가 소유한다.
