ROUTE / 01

현재 상태에서 작업 찾기

“무엇을 만들까?”보다 “지금 어느 경계에 있나?”를 먼저 확인한다. 시작 위치와 완료 증거가 다음 행동을 결정한다.

새 기능을 계획한다

INTENT

해석이 갈릴 수 있는 요구사항을 수용기준과 작은 태스크로 고정한다.

시작 위치develop / git 없음
사용할 절차plan
완료 증거승인된 spec
다음feature-add

기능을 구현한다

BUILD

신규 기능은 RED→GREEN→Refactor, 기존 기능은 바뀔 동작만 RED로 만든다.

시작 위치develop / feature/*
사용할 절차feature-add · feature-modify
완료 증거tests green + commit
다음verification

실패 원인을 찾는다

DIAGNOSE

원인을 모르면 수정부터 하지 않는다. 재현하고 최대 3개 가설을 판별한다.

시작 위치현재 브랜치
사용할 절차systematic-debugging
완료 증거root cause / 미확정 근거
다음feature-modify

완료 주장을 검증한다

VERIFY

현재 worktree·HEAD·CI SHA에 대해 새로 실행한 명령으로만 완료를 판정한다.

시작 위치어느 브랜치든
사용할 절차verification-before-completion
완료 증거VERIFIED / NOT VERIFIED
다음feature-merge

PR·릴리즈로 보낸다

SHIP

feature-merge는 develop까지, release-check → release는 main 태그와 develop back-merge까지 소유한다. 모든 PR은 review·CI·thread·status 게이트를 지난다.

시작 위치feature/* · fix/* · develop
사용할 절차feature-merge · release-check · release
완료 증거merged PR · tag · health
우회 금지raw gh · direct push
FLOW / 02

한 기능이 이동하는 레일

각 칸은 실제 순서다. 앞 칸의 증거가 없으면 다음 칸으로 이동하지 않는다.

01 / INTENT

의도 고정

무엇을·왜, 정상·예외·경계, 비범위를 승인한다.

proof: docs/specs/*.md
02 / RED

실패 고정

구현 전 테스트가 올바른 이유로 실패하는지 확인한다.

proof: exact command + exit
03 / GREEN

최소 구현

계약을 통과시키는 최소 변경 뒤 중복을 정리한다.

proof: focused tests green
04 / VERIFY

현재 상태 검증

전체 품질·diff·worktree·HEAD를 새 증거로 대조한다.

proof: VERIFIED
05 / REVIEW

반증과 CI

AI 리뷰, 사람 승인(팀), 스레드, required CI를 닫는다.

proof: all gates green
06 / SHIP

머지·태그

래퍼로 머지하고 릴리즈면 태그·back-merge·health까지 본다.

proof: SHA continuity
EVIDENCE / 03

추측을 멈추는 두 스킬

systematic-debugging은 “왜 실패했나”를, verification-before-completion은 “정말 끝났나”를 닫는다.

systematic-debugging

진단 요청이면 작업트리를 바꾸지 않는다. 수정 요청이어도 근본 원인이 확인되기 전에는 패치하지 않는다.

  1. 기대값·실제값·재현 명령·종료 코드 고정
  2. 확인된 사실과 추론 분리
  3. 최대 3개 가설과 판별 실험
  4. 근본 원인 또는 미확정 경계 보고
systematic-debugging으로 이 CI 실패의 원인만 찾아줘. 아직 수정하지 마.

verification-before-completion

실패·미실행·SHA 불일치는 완료가 아니다. 검증 스킬은 실패를 직접 고치거나 머지 권한을 만들지 않는다.

  1. 검증할 주장을 관찰 가능한 증거에 매핑
  2. worktree와 HEAD SHA 구분
  3. 명령과 종료 코드를 새로 수집
  4. VERIFIED / NOT VERIFIED로 fail-closed 판정
이 브랜치가 PR 준비됐다는 주장을 현재 HEAD의 새 증거로 검증해줘.
GATES / 04

머지는 명령이 아니라 게이트다

PR 생성과 머지는 저장소 래퍼가 소유한다. AI 리뷰가 사람 승인을 대신하지 않고, 체크런 성공이 미해결 스레드를 대신하지 않는다.

AI 리뷰변경분을 반증하고 지적의 사실관계를 확인
Thread 0답변을 남기고 모든 review thread resolve
사람 승인팀 모드만 적용, 솔로 표준은 승인요건 0
Required CIquality · secret scan · project checks
SHA 머지외부 status와 HEAD 일치 뒤 pr-merge.sh
릴리즈는 여기서 끝나지 않는다. release-check 품질·보안·DB 사전검증 → release main PR → 태그 → develop back-merge PR → 배포 후 health 순서다.
RECOVERY / 05

막혔을 때 우회하지 않기

차단 메시지는 다음 행동을 알려주는 라우팅 정보다. 가드를 끄거나 테스트를 삭제하면 원인은 남는다.

차단 메시지·증상의미다음 행동
main/develop 직접 커밋 금지보호 브랜치에서 변경 중feature/* 또는 fix/* 브랜치
feature 브랜치는 plan 필요기능 의도와 수용기준 미승인plan으로 spec 작성·승인
raw gh pr create/merge 금지PR 게이트 누락 방지pr-create · feature-merge · release
테스트 실패현재 구현이 계약을 충족하지 않음systematic-debugging으로 원인 판별
CI 성공, 머지 불가스레드·승인·mergeability 등 다른 게이트 미충족pr-review-gate 상태 확인
skill·hook이 보이지 않음설치 cache·hook trust·repo sync 문제 가능harness-doctor.sh --repo .

차단별 상세 복구법 보기

TOOLS / 06

UI는 달라도 완료 조건은 같다

스킬이 화면에서 어떻게 보이는지는 실행 표면의 차이다. 테스트·래퍼·CI·branch protection은 동일하다.

Claude Code

slash skill과 역할별 agent가 명시적으로 보일 수 있다. hooks와 plugin 원본 계약을 사용한다.

Codex

현재 agent가 읽은 SKILL.md를 직접 수행할 수 있다. overlay·adapter가 런타임 차이를 매핑한다.

설치·도구별 설정은 온보딩, 책임 경계는 AI 협업 가이드, PR 기준은 코드리뷰 정본을 따른다.

DONE / 07

완료라고 말하기 전

모두 체크할 수 없으면 작업은 끝나지 않았다.

  • 요청한 범위만 변경했다
  • 정상·예외·경계를 테스트했다
  • 전체 품질 명령이 exit 0이다
  • 현재 worktree와 HEAD를 구분했다
  • AI 리뷰 지적과 스레드를 닫았다
  • required CI와 승인 요건을 통과했다
  • CI·배포 SHA가 대상 HEAD와 같다
  • 릴리즈면 태그·back-merge·health를 확인했다
