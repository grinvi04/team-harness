# /systematic-debugging — 증거 기반 근본 원인 디버깅

수정부터 시작하지 말고 **재현 → 사실 → 가설 → 판별 → 원인 확정 → 최소 수정** 순서를 지킨다.
증상을 잠시 숨기는 패치가 아니라 증상을 만든 인과 메커니즘을 찾는다.

## 완료 기준

| 모드 | 완료 조건 |
|---|---|
| 진단 전용 | 재현 결과, 확인된 근본 원인 또는 미확정 사유, 다음 증거를 보고하고 작업트리를 바꾸지 않음 |
| 수정 포함 | RED 회귀 계약, 최소 수정, 대상 검사와 전체 품질 GREEN, 외과적 diff를 보고함 |

근본 원인을 확정하지 못했으면 진단도 수정도 완료로 표현하지 않는다.

## Phase 0 — 모드와 안전 경계 고정

1. repo의 `AGENTS.md`와 작업 대상 stack rule을 읽어 실제 테스트·빌드·품질 명령을 확인한다.
2. 사용자의 요청을 구분한다.
   - **진단 전용:** 원인과 근거만 보고한다. 파일을 수정하지 않고 브랜치·커밋·PR도 만들지 않는다.
   - **수정 포함:** Phase 3 원인 확정 게이트를 통과한 뒤에만 Phase 4로 간다.
3. `git status --short --branch`로 시작 상태를 기록한다. 관련 없는 미커밋 변경이 있으면 수정 모드는 중단하고 범위를 확인한다. 진단 전용은 읽기만 계속할 수 있다.
4. 운영 DB·운영 인프라 변경, destructive probe, 시크릿 출력은 진단 수단으로 사용하지 않는다.

## Phase 1 — 같은 실패를 재현

다음 재현 기록을 먼저 만든다.

| 항목 | 기록할 내용 |
|---|---|
| 기대값 | 정상이라면 관찰돼야 할 동작 |
| 실제값 | 실제로 관찰된 동작과 최초 실패 지점 |
| 재현 | 정확한 입력·환경·재현 명령·종료 코드 |
| 범위 | 항상/간헐, 특정 OS·브랜치·데이터·SHA 여부 |

- 기억이나 이전 로그가 아니라 현재 상태에서 안전한 재현 명령을 새로 실행한다.
- 간헐 실패는 반복 횟수와 성공/실패 분포를 기록한다. 재현되지 않으면 환경 차이와 추가 관측 지점을 보고하고 추측 수정으로 넘어가지 않는다.
- 오류가 연쇄되면 마지막 stack trace보다 **처음 잘못된 값이나 경계**를 찾는다.

## Phase 2 — 사실을 모으고 경계를 좁힘

관찰한 **사실**과 아직 검증하지 않은 **추론**을 분리해 적는다.

1. 실패 경로의 입력 → 변환 → 출력 흐름을 역추적한다.
2. 마지막 정상 지점과 첫 실패 지점 사이의 가장 작은 경계를 찾는다.
3. 정상 사례와 실패 사례를 같은 축으로 비교한다. 한 번에 변수 하나만 바꾼다.
4. 라이브러리·프레임워크 동작이 쟁점이면 학습 기억 대신 설치된 버전의 문서·소스를 읽는다.
5. 로그·mock·fixture가 실제 경계를 가려 false positive를 만들지 않는지 확인한다.

## Phase 3 — 가설을 판별하고 근본 원인을 확정

증거로 설명 가능한 가설을 **최대 3개**만 우선순위로 둔다.

| 순위 | 가설 | 지지/반박 사실 | 판별 실험 | 결과 |
|---|---|---|---|---|
| 1 | 가장 가능성 높은 원인 | 현재 증거 | 이 가설만 참이면 달라질 관찰 | 확인/기각 |

- 각 가설에 하나의 판별 실험을 붙이고, 가장 작고 안전한 실험부터 수행한다.
- 여러 가설을 한 패치로 동시에 시험하지 않는다. 무엇이 원인이었는지 알 수 없게 된다.
- 기각된 가설의 변형을 반복하지 말고 새 증거가 있을 때만 우선순위를 갱신한다.

### 원인 확정 게이트

다음을 모두 만족할 때만 근본 원인을 확정한다.

1. 원인이 기대값과 실제값의 차이를 만드는 인과 경로를 설명한다.
2. 판별 실험 결과가 원인을 지지하고 경쟁 가설을 기각한다.
3. 수정해야 할 가장 작은 경계와 회귀 검사 위치를 특정할 수 있다.

근본 원인을 확정하기 전에는 코드를 수정하지 않는다. 세 가설이 모두 기각되거나 증거가 부족하면 원인을 확정하지 못했다고 보고하고, 다음으로 필요한 관측을 제시하며 추측으로 수정하지 않는다.

## Phase 4 — 수정 요청일 때만 RED → GREEN

원인 확정 게이트를 통과했고 사용자 요청에 수정이 포함될 때만 실행한다.

1. 브랜치와 작업트리를 보존한다.
   - 이미 `feature/*`·`fix/*`·`hotfix/*` 작업 브랜치면 그대로 유지한다.
   - 보호 브랜치이고 작업트리가 깨끗하면 repo 규약에 맞는 `fix/*` 브랜치를 만든다.
   - 관련 없는 미커밋 변경이 있으면 중단하고 임의 stash·커밋·브랜치 전환을 하지 않는다.
2. 공개 동작에는 자동 회귀 테스트를, CI·빌드·설정 장애에는 가장 작은 실행 가능 검사를 만들고 **수정 전 RED**를 확인한다. 문법·import 오류가 아니라 원래 증상 때문에 실패해야 한다.
3. 회귀 계약을 잠근다. 이후 테스트를 삭제·약화·skip해 통과시키지 않는다.
4. 기존 동작 수정은 `feature-modify`의 RED→GREEN·전체 품질 계약을 적용한다. 근본 원인만 제거하는 최소 코드로 GREEN을 만든다.
5. 같은 수정 루프가 3회 실패하면 패치를 더 쌓지 말고 Phase 2로 돌아가 가설을 재검토한다.

## Phase 5 — 원래 증상과 전체 회귀 검증

1. 새 회귀 검사와 원래 재현 명령을 각각 다시 실행한다.
2. `AGENTS.md`의 전체 품질 명령을 실행한다.
3. `git diff --check`, `git diff --stat`, `git diff`로 요청 밖 변경과 테스트 약화가 없는지 확인한다.
4. 간헐 실패였다면 Phase 1과 같은 반복 조건으로 재검증한다.

하나라도 실패하면 수정 완료로 보고하지 않는다. 실패 증거와 현재 가설을 가지고 해당 Phase로 돌아간다.

## 결과 보고

다음을 짧게 보고한다.

- 재현 명령과 결과
- 확정한 근본 원인과 판별 증거, 또는 미확정 사유
- 수정 모드라면 RED → GREEN 증거와 변경 파일
- 전체 품질 결과와 아직 검증하지 못한 한계
