# product-direction 스펙

## 1. 목표 & Why

Team Harness가 실행 플랫폼의 일반 기능을 반복 구현하지 않고 GitHub 기반 정책·증거·감사·delivery 강제에
집중하도록 제품 경계를 정본으로 남긴다. **성공 기준: 신규 기능 제안이 소유·연결·위임 중 하나로 일관되게
판정되고, README와 AI 작업 계약이 같은 정본을 가리킨다.**

## 2. Scope

- **In:** 제품 정체성, 직접 소유/플랫폼 위임 경계, 설계 원칙, 기능 판단 기준, 우선순위 로드맵.
- **Out (Non-goals):** 기존 skill·hook 제거, 호환 코드 즉시 리팩터링, 외부 프로젝트 비교·평가, 버전 bump.

## 3. 기능 요구사항 + 수용기준 (= 테스트 계약)

- **AC-1 (정본):** WHEN 제품 방향을 확인할 때, 단일 문서가 정체성·소유·위임 경계를 함께 설명해야 한다.
- **AC-2 (작업 게이트):** WHEN team-harness에 기능을 추가·변경할 때, AGENTS는 제품 방향 정본과 신규 기능
  판단 질문을 먼저 적용하도록 지시해야 한다.
- **AC-3 (발견성):** WHEN README에서 프로젝트를 처음 볼 때, 제품 방향과 정본 링크를 빠르게 찾을 수 있어야 한다.
- **AC-4 (지속성):** WHEN 향후 플랫폼 기능이 추가될 때, native-first·thin adapter·outcome parity 원칙으로
  소유·연결·위임을 재판정할 수 있어야 한다.
- **AC-5 (실행성):** WHEN 다음 작업을 선택할 때, 공개 안전성·중복 감사·제품화·외부 검증 순서가 명시돼야 한다.

## 4. 제약 / 비기능

- 특정 외부 프로젝트와의 비교나 경쟁 문구 없이 장기적으로 안정적인 원칙만 기록한다.
- 플러그인 동작·소비 repo 자산은 바꾸지 않으므로 버전은 유지한다.

## 5. 경계 / Do-Not

- ✅ 해도 됨: README·AGENTS·결정 기록에서 방향 정본을 참조.
- ⚠️ 먼저 물어봐: 기존 skill·hook·호환 어댑터 제거 또는 실행 계약 변경.
- 🚫 절대 금지: 플랫폼 기본 기능을 team-harness의 독자 기능으로 과장하거나 검증되지 않은 강제력을 주장.

## 6. Open Questions

없음. 사용자가 제품 방향과 특정 외부 프로젝트 언급 제외를 승인했다.

## 7. 기술 접근 (HOW)

- `docs/product-direction.md`를 제품 방향의 단일 출처로 만든다.
- `AGENTS.md`에 기능 제안 전 방향 게이트를 추가한다.
- `README.md`에는 짧은 포지셔닝과 정본 링크만 두고 상세를 중복하지 않는다.
- `docs/decisions.md`에 방향 전환의 이유와 향후 판단 규칙을 기록한다.
- 문서 링크와 필수 섹션을 shell 기반 계약으로 검증한다.
