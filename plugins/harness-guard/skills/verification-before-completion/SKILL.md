# /verification-before-completion — 완료 주장 전 증거 게이트

“될 것이다”와 “현재 됐다”를 구분한다. 완료라는 말보다 먼저, **바로 지금의 상태를 증명하는 증거**를 수집한다.

## 완료 기준

1. 검증할 주장을 관찰 가능한 증거와 1:1로 매핑한다.
2. 현재 작업트리와, 변경이 커밋됐다면 정확한 HEAD SHA를 기준으로 증거를 새로 수집한다.
3. 각 항목을 `PASS / FAIL / UNVERIFIED / SKIP`으로 판정하고 명령·종료 코드·대상 SHA를 남긴다.
4. 필수 항목에 `FAIL`이나 `UNVERIFIED`가 하나라도 있으면 완료로 판정하지 않는다.

## Phase 0 — 호출 맥락과 권한 고정

1. repo의 `AGENTS.md`와 작업 대상 stack rule에서 테스트·빌드·품질 명령을 읽는다.
2. 호출 맥락을 구분한다.
   - **상위 workflow에서 호출:** 검증 결과만 돌려준다. 호출 workflow가 최종 판정을 소유한다.
   - **직접 호출:** 확인·검증 결과만 보고한다. 실패나 미확인이면 파일을 수정하지 않고 보고 후 중단한다.
3. 이 스킬은 수정·커밋·PR·머지·릴리즈 권한을 새로 만들지 않는다. 실패 수정은 사용자가 맡긴 기존 workflow의 범위에서만 수행한다.
4. 운영 DB나 운영 인프라를 변경하지 않는다. destructive probe나 실제 고객 요청 생성도 완료 증거로 사용하지 않는다.

## Phase 1 — 주장을 증거 계획으로 변환

모호한 완료 주장을 관찰 가능한 증거로 바꾼다.

| 주장 | 우선 증거 |
|---|---|
| 버그가 해결됨 | 원래 재현 명령 + 잠긴 회귀 검사 |
| 기능이 동작함 | 수용기준별 대상 테스트 + 핵심 e2e |
| 회귀가 없음 | `AGENTS.md` 전체 품질 명령 |
| 변경 범위가 정확함 | `git diff --check` + stat + 전체 diff |
| PR·머지 준비 완료 | 현재 HEAD와 PR head SHA 일치 + required CI·리뷰·mergeability |
| 릴리즈·배포 준비 완료 | release gate + 배포 대상 SHA 일치 + 읽기 전용 health 상태 |
| 문서·설정 완료 | 구문·링크·소비자 경로 검사 |

- 검증 대상이 없는 항목을 억지 명령으로 꾸미지 않는다. 적용 불가 항목은 `SKIP`으로 표시하고 이유를 쓴다.
- “테스트 통과”처럼 범위가 불명확하면 대상 테스트인지 전체 품질인지 먼저 구체화한다.

## Phase 2 — 현재 상태의 새 증거 수집

1. `git status --short --branch`로 현재 작업트리 상태를 기록한다.
2. 커밋된 변경이면 `git rev-parse HEAD`로 HEAD SHA를 기록한다. 미커밋 변경은 SHA가 증명하지 못하므로 diff를 별도 증거로 취급한다.
3. 계획한 검증 명령과 종료 코드를 지금 새로 수집하고 핵심 출력을 기록한다. 이전 세션의 기억이나 오래된 로그만으로 판정하지 않는다.
4. CI와 배포 증거는 reported SHA가 현재 HEAD와 일치하는지 확인한다. 다른 commit·branch의 green 결과는 현재 변경의 증거가 아니다.
5. 외부 상태를 읽을 수 없으면 성공으로 추정하지 말고 `UNVERIFIED`로 둔다.

## Phase 3 — 증거 자체를 반증

PASS를 선언하기 전에 false positive 가능성을 공격한다.

- 대상 테스트가 실제 변경 동작을 단언하는지 확인한다. 존재·truthy만 보는 약한 테스트나 구현을 그대로 베낀 테스트는 증거로 부족하다.
- 버그 수정은 가능하면 수정 전 RED가 원래 증상 때문에 실패했다는 기록을 대조한다.
- guard·차단·권한처럼 거부가 본질이면 허용 사례만 보지 말고 안전한 반례·우회 입력을 시험한다.
- 전체 품질 명령이 실패 테스트를 skip하거나 다른 디렉터리만 검사하지 않는지 확인한다.
- diff에 요청 밖 파일, 테스트 약화, 생성물, 시크릿이 섞이지 않았는지 확인한다.

반증을 위해 테스트를 수정하거나 운영 상태를 바꾸지 않는다.

## Phase 4 — 선택적 독립 검증

위험이 크거나 판단이 갈리는 주장만 verifier에 선택적 독립 반증을 맡긴다.

- verifier에는 현재 주장, raw diff, 명령 결과처럼 원본 증거를 제공한다. 기대 결론을 주입하지 않는다.
- verifier는 증거 수집과 반증만 한다. verifier가 파일을 수정·커밋·머지하지 않으며 완료 판정을 대신하지 않는다.
- 독립 검증 결과가 충돌하면 현재 호출 workflow가 차이를 해소할 때까지 `UNVERIFIED`로 둔다.

## Phase 5 — fail-closed 판정

| 상태 | 의미 | 완료 주장 가능 여부 |
|---|---|---|
| PASS | 현재 상태를 직접 증명함 | 모든 필수 항목 PASS일 때만 가능 |
| FAIL | 명령 실패 또는 반례 발견 | 불가 |
| UNVERIFIED | 실행 못 함, 증거가 stale·불충분·다른 SHA | 불가 |
| SKIP | 실제로 적용 불가하며 이유 기록 | 다른 필수 항목으로 판정 |

실패나 미확인 항목이 있으면 완료로 판정하지 않는다. 상위 workflow 호출이면 실패 증거와 함께 해당 단계로 돌려보내고, 직접 호출이면 무수정 상태로 보고하고 중단한다.

## 결과 보고

다음 표를 사용한다.

| 주장 | 증거 명령·대상 | 결과 | 판정 |
|---|---|---|---|
| 예: 회귀 없음 | `test-command` (worktree/HEAD SHA) | exit 0/1 + 핵심 출력 | PASS/FAIL |

마지막에 전체 판정(`VERIFIED` 또는 `NOT VERIFIED`)과 아직 확인하지 못한 한계를 명시한다. 실행하지 않은 항목을 실행한 것처럼 말하지 않는다.
