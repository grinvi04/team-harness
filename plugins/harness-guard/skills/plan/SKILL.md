# /plan — AI 실행용 스펙·플랜 정리 (계획 전용, git 무관)

**사용법**: `/plan <feature-name> "<무엇을·왜>"`
예) `/plan checkout "장바구니 결제 — 재고 차감·결제 연동·주문 생성"`

> 푸는 문제는 **드리프트** — AI가 그럴듯하지만 의도와 어긋난 코드를 확신있게 만드는 것.
> `/plan`의 가치는 **코드 전에 의도·제약·수용기준을 박제**해 재작업 루프를 없애는 데 있다(문서량 아님).
> **`/plan`은 분석·계획만 한다 — git을 건드리지 않는다**(브랜치·커밋 없음). 브랜치 생성·커밋·구현은 개발 단계(`/feature-add`)가 소유한다.
> 산출물은 **파일 1장**(`docs/specs/<feature-name>.md`) — 스펙→플랜→태스크 통합. 솔로/소팀은 파일을 쪼개지 않는다.

> **흐름**: `/plan`(계획·승인, git 무관) → 개발: **`feature/<기능>` 한 브랜치**에서 태스크별 `/feature-add`(TDD·태스크당 원자적 커밋) → `/feature-merge`(한 PR). *한 기능 = 한 브랜치 = 한 PR.*

> **스택 의존 값은 repo의 `AGENTS.md`에서 읽는다**(빌드·테스트 명령, 소스 디렉터리, 프론트 유무). 하드코딩 금지.
> **비협상 규칙은 `CLAUDE.md` 4원칙**(단순함 우선·외과적 수정·목표기반 실행·생각 먼저)을 그대로 따른다 — 별도 constitution을 만들지 않는다.

---

## 언제 쓰는가

- ✅ **`/plan` 사용**: 여러 동작·도메인이 얽힌 기능, 의도/범위가 불명확한 기능, 한 RED-GREEN으로 안 끝나는 규모.
- 🚫 **생략하고 바로 `/feature-add`**: 단일·자명한 기능, typo·1줄 수정, 동작이 이미 명확한 작은 변경. (스펙을 위한 스펙 = 과잉)

---

## 완료 기준 (Definition of Done) — 통과까지 루프한다

| # | 기준 | 검증 방법 |
|---|---|---|
| 1 | `[NEEDS CLARIFICATION]` 0개 — 모호점 전부 사람에게 물어 해소 | 스펙 내 마커 검색 = 0건 |
| 2 | 각 수용기준이 테스트화 가능(EARS/Given-When-Then, 정상·예외·경계, AC-ID 부여) | 기준마다 "이걸 어떻게 테스트?"에 1답 |
| 3 | Out-of-scope(비목표)·Do-Not 경계 명시 | 해당 섹션 존재 |
| 4 | 각 태스크가 작고(한 RED-GREEN=원자적 커밋 단위) AC 참조·파일경로·검증커맨드·의존성 보유 | 태스크 표 대조 |
| 5 | 사람 승인 1회 완료 | Phase 5 게이트 |

## Phase 0 — 진입 판정 + 컨텍스트 (git 없음)

1. 위 "언제 쓰는가"로 `/plan` 적합성 판정 — 생략 대상이면 `/feature-add` 안내 후 종료.
2. `AGENTS.md`를 읽어 스택·디렉터리·프론트 유무·빌드/테스트 명령 파악.
3. **Claude Code plan mode 진입** — 코드베이스를 읽기 전용으로 보호하고 사람 승인 없이 편집 도구를 쓰지 못하게 한다. Plan mode는 Phase 5 사람 승인(ExitPlanMode)까지 유지된다.

> ⛔ 브랜치를 만들지 않는다. `/plan`은 현재 위치(보통 develop)에서 **읽기·계획만** 한다.
> Plan mode가 이를 메커니즘으로 강제한다 — prose 규칙이 아니다.

---

## Phase 1 — 탐색 (read-only)

코드베이스를 **읽기만** 한다 — 재사용할 기존 패턴·코드, 영향 모듈, 제약(스키마·권한·외부 연동)을 파악.
**추측 금지**: 불명확한 점은 조용히 가정하지 말고 `[NEEDS CLARIFICATION: 질문]`으로 표면화한다(미지정 요구는 회귀 확률 ~2배).

> **Preflight (계획 전 선독, 필수)**: 이 기능이 건드리는 ① 해당 team-harness **표준 섹션**과
> ② **설치된 라이브러리의 실제 API/소스**(학습데이터가 아니라 `node_modules`의 설치본 문서·
> `src/components/ui/*` 등 repo에 들어온 실제 코드)를 *계획을 쓰기 전에* 읽는다. 알아낸 함정·제약을
> 플랜(§7 HOW)·태스크에 적는다. (근거: 'This is NOT the Next.js you know — node_modules 문서를 읽어라' —
> 버전이 바뀌면 API도 바뀐다. 기억으로 계획하지 말 것.)

## Phase 2 — 스펙 작성 (WHAT/WHY만, `docs/specs/<feature-name>.md`)

> 선언적으로 — **무엇을·왜**만. 기술스택·API·코드구조(HOW)는 여기 쓰지 않는다(기술이 바뀌어도 스펙은 안정).

아래 템플릿으로 작성한다:

```markdown
# <feature-name> 스펙

## 1. 목표 & Why
<무엇을·왜 1~3문장>. **성공 기준(측정 가능): <한 줄>**

## 2. Scope
- **In:** <포함>
- **Out (Non-goals):** <명시적 비범위 — 누락으로 추론하게 두지 말 것>

## 3. 기능 요구사항 + 수용기준 (= 테스트 계약)
각 기준에 AC-ID. EARS/Given-When-Then으로 한 해석만 남게. 정상·예외·경계를 분리.
- **AC-1 (정상):** WHEN <트리거>, the system SHALL <관찰 가능한 응답>.
- **AC-2 (경계):** WHILE <상태/경계조건>, the system SHALL <응답>.
- **AC-3 (예외):** IF <오류조건> THEN the system SHALL <오류 응답·코드>.

## 4. 제약 / 비기능 (해당될 때만, 솔로면 1~2줄)
<성능·보안·a11y 등 — 정량 임계값으로>

## 5. 경계 / Do-Not (3단계)
- ✅ 해도 됨: <범위 내 자유 영역>
- ⚠️ 먼저 물어봐: <영향 큰 결정>
- 🚫 절대 금지: <시크릿 커밋·건드리면 안 되는 경로/기능>

## 6. Open Questions
- [NEEDS CLARIFICATION: <미지정·불확실 항목>]   ← Phase 5 전에 0개로
```

⛔ 안티패턴: 모호한 수용기준("잘 동작하게"), 구현 디테일 누수(클래스명·엔드포인트·DB테이블), 전 기능을 한 문서에(1~4페이지·기능 단위 상한), 지시 과적재.

## Phase 3 — 플랜(HOW) 섹션 (같은 파일 하단에 추가)

```markdown
## 7. 기술 접근 (HOW)
- 아키텍처/접근 개요 + 기술 선택 근거(스택은 AGENTS.md 따름)
- 영향받는 파일/모듈
- 테스트 전략: 어느 AC를 어느 레벨(단위/통합/e2e) 테스트로 — AC ↔ 테스트 1:1
```
설계 lens(규모에 맞게): 의존성 규칙(도메인이 IO/프레임워크에 의존하지 않게), SOLID는 judicious(추측성 추상화 금지). repo가 `clean-architecture.md`를 따르면 그 경계 존중.

## Phase 4 — 태스크 분해 (롤백 가능하게)

각 태스크 = **수직 슬라이스 + 독립 + 한 RED-GREEN 사이클** = **원자적 커밋 1개**(되돌림 단위). 의존성 순서로 정렬.

```markdown
## 8. 태스크 (test-first 순서)
| # | 태스크 | AC 참조 | 대상 파일 | 검증(이 명령 exit 0) | 의존 | [P] |
|---|---|---|---|---|---|---|
| 1 | <작은 동작 단위> | AC-1,AC-3 | path/to/x | <테스트/빌드 명령> | — | |
| 2 | ... | AC-2 | ... | ... | #1 | |
| 3 | ... | AC-4 | ... | ... | — | [P] |
```
- 순서: contracts → tests → implementation. 각 태스크의 수용기준이 곧 `/feature-add` Phase 2의 RED 테스트 계약 입력이 된다.
- **롤백 설계**: 태스크 = 원자적 커밋이라 **커밋 1개 = 롤백 1단위**. **독립 태스크(`[P]`, 의존 없음)** 는 `git revert <그 커밋>`로 단독 롤백 가능. **의존받는 태스크**(뒤 태스크가 그 위에 쌓임)는 단독 롤백 대신 **fix-forward**(DB 마이그레이션 forward-only 원칙과 동일 — 앞으로 고친다). → 태스크를 작게·의존을 명시할수록 롤백이 안전해진다.
- ⛔ 거대 태스크 금지(여러 RED-GREEN을 한 태스크로 묶으면 실행 품질 급락 + 롤백 단위가 비대해짐).

## Phase 5 — 승인 게이트 (사람, 필수 — /plan은 여기서 끝)

1. `[NEEDS CLARIFICATION]`가 남아 있으면 **사람에게 질문**하고 답을 스펙에 반영(0개가 될 때까지). 추측으로 채우지 않는다.
2. 스펙(목표·Scope·수용기준)·태스크 목록을 요약 제시하고 **사람 승인을 받는다.**
   > 잘못된 방향으로 자율 실행되는 것을 막는 가장 싼 결함 차단 지점. 솔로도 **계획 1회 승인**은 필수(스텝별 승인은 과잉).
3. 승인 시 **Claude Code plan mode 종료(ExitPlanMode)** — 이후 개발 단계(`/feature-add`)에서 편집 도구 사용이 허용된다.
4. `/plan`은 git을 건드리지 않고 **승인된 계획 산출물에서 종료**한다.

> 스펙 파일(`docs/specs/<feature-name>.md`)은 계획 산출물 — **`/plan`은 커밋하지 않는다.** 개발 단계가 브랜치를 만들 때 코드와 함께 커밋한다.

---

## Phase 6 — 핸드오프 (개발 단계로; git은 여기부터 — `/plan` 밖)

`/plan`은 git을 쓰지 않으므로, 승인 후 개발로 넘긴다(아래는 개발 단계가 수행):
1. `feature/<기능>` 브랜치를 **한 번** 만든다 — 첫 태스크의 `/feature-add`가 develop에서 생성.
2. 태스크 표 **순서대로** 각 태스크를 `/feature-add <task> "<수용기준 AC-N 반영 설명>"`로 개발 — **같은 `feature/<기능>` 브랜치에서**(한 기능=한 브랜치), TDD(RED→GREEN→Refactor), **태스크당 원자적 커밋**(체크포인트=롤백 단위).
3. 회귀/오류 시: **독립 태스크**면 `git revert <커밋>`, **의존 태스크**면 fix-forward(앞으로 고침).
4. 전 태스크 GREEN + 스펙 수용기준 전부 충족 → 사용자 확인 후 `/feature-merge`(한 PR).

> ⚠️ 큰 작업을 한 세션에 몰아 실행하지 않는다(컴파운딩 에러·~35분 자율 절벽). 태스크 단위로 끊고 커밋 체크포인트를 남긴다.
