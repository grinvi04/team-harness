# AI 협업 가이드

이 하네스의 존재 이유 — 5–10명이 각자의 방식으로 AI를 쓰면 코드 편차가 AI 도입 전보다 커진다.
도구가 무엇이든(Claude Code·Codex 개발 경로 지원, 기타 도구는 공통 계약 적용) 아래 규칙은 공통이다.

## 대원칙

> **AI가 만든 결과물의 책임은 그것을 PR에 올린 사람에게 있다.**

- AI는 작성자(author)이고 당신은 그 작성자를 검수하는 시니어다 — 검수 없는 머지는 본인 책임
- **이해하지 못한 코드는 머지하지 않는다** (설명 가능성 원칙): 리뷰에서 "왜 이렇게 했나"에
  답하지 못하면 그 코드는 아직 당신 것이 아니다
- AI 사용을 숨기지 않는다 — 커밋에 `Co-Authored-By: <도구>` 표기 유지

## 작업을 맡기는 방법

- **명세를 포함해 위임한다**: 목표 + 제약(이 표준 문서들) + 검증 기준(통과해야 할 테스트)을
  첫 지시에 담는다. "알아서 해줘"는 토큰 낭비와 재작업의 지름길
- **한 PR = 한 작업**: AI는 시키면 끝없이 만든다 — 요청 범위 밖 리팩터링·기능 추가는 거절하고 분리
- **병렬 에이전트는 worktree로 격리한다**: 여러 에이전트가 같은 체크아웃을 공유하면 브랜치 전환·reset이
  서로의 미커밋 작업을 clobber한다(실제로 동시 작업 중 미커밋 변경이 유실된 사고에서). 동시에 코드를 수정하는 에이전트는
  각자 git worktree(또는 격리된 작업공간)에서 돌리고, 머지는 순차로(뒤처진 브랜치는 base 최신화 후)
- **대량 파괴적 작업(브랜치·파일 일괄 삭제)은 실행 직전 독립 재검증한다**: 분류·필터 로직을 믿고 바로
  지우지 않는다 — ad-hoc 셸 명령은 셸 차이로 조용히 오분류될 수 있다(스크립트는 `#!/usr/bin/env bash`라
  무관하나, 대화형·도구 셸이 zsh면 **unquoted 확장을 단어분할하지 않아** `for x in $list`가 리스트를
  통째로 처리해 에러 없이 틀린다 — 리스트 순회는 `while read`로). 실제로 브랜치 정리 스캔이 전부
  '미머지'로 오판했다가 삭제 직전 재검증(`git merge-base --is-ancestor`)으로 바로잡은 사고에서.
- 막연한 탐색보다 **TDD 루프에 태운다**: 실패하는 테스트를 먼저 (사람이 또는 AI가) 확정하고,
  통과를 목표로 작업시키면 결과 검증이 자동화된다
- 규약은 repo의 `AGENTS.md`가 단일 출처 — AI에게 별도 장문 지시 대신 AGENTS.md를 채워라
- **생성 문서는 repo에 커밋한다**: AI가 만든 계획·설계 문서(`/plan` 스펙, `/milestone` 추적,
  설계 결정 등)는 도구 로컬 캐시(예: `~/.claude/plans`, `~/.codex`)에 두지 말고 프로젝트 `docs/` 아래에 커밋한다.
  로컬 캐시는 노트북·도구·세션이 바뀌면 유실 — repo에 있어야 누가·어디서 이어받아도 일관된다.
  각 repo의 `AGENTS.md`는 자기 `docs/` 경로(스펙·마일스톤)를 명시한다.
- **프로젝트 상태·결정·도메인 지식은 로컬 AI 메모리에 두지 않는다**:
  도구 로컬 메모리(예: Claude `~/.claude`, Codex `~/.codex`)는 **다른 PC·세션·사람이 못 보고 유실되므로
  팀 지식 저장소로 쓰지 않는다.** 로컬 메모리는 팀 공유 불필요한 *개인 작업습관·행동 교정*에만 최소로 쓴다.

| 기록 유형 | 정본 위치 | 원칙 |
|---|---|---|
| 플랜·스펙 | `docs/specs/` | `/plan` 산출물은 repo 문서로 남기고, 개발 브랜치에서 코드와 함께 커밋한다. |
| 백로그·할 일 | GitHub Issues + GitHub Milestone(`/milestone`) | PR 링크·담당·상태 추적이 필요한 항목은 Issues가 정본이다. 단순 `docs/backlog.md`는 stale 위험이 있어 보조 색인 이상으로 쓰지 않는다. |
| 마일스톤 정의·진행률 | GitHub Milestone + `docs/milestones/` | `/milestone` 스킬이 생성·갱신하는 위치를 따른다. |
| 작업로그·완료 기록 | git 히스토리 + CHANGELOG/릴리즈노트 | "무엇을 했나"는 커밋·PR·릴리즈노트가 정본이다. 별도 worklog는 중복이므로 만들지 않는다. |
| 결정·도메인 지식·이유 | `docs/decisions.md` | 비자명한 선택·제약·"왜 이렇게 했나"만 누적한다. |
| 로컬 AI 메모리 | 개인 습관·행동교정만 | 프로젝트 상태, 백로그, 작업로그, 결정, 도메인 지식 저장 금지. |

### Skill 실행과 가시성

Claude Code의 slash skill과 Codex의 skill 로딩은 UI가 다르다. Claude는 명시적인 slash surface를 사용할 수
있지만 Codex는 `SKILL.md` 지침을 현재 agent가 직접 수행하므로 별도 tool-call이 표시되지 않을 수 있다.
공통 계약은 UI가 아니라 결과다: 동일 수용 기준, 같은 PR wrapper, 같은 리뷰·CI 게이트를 따른다.

사용자가 실제 적용 여부를 확인할 수 있도록 AI는 skill을 적용하는 첫 업데이트와 Git flow 단계 전환 때
**적용 skill과 현재 phase**를 짧게 표시한다. Codex에 존재하지 않는 `Skill 도구` 호출을 요구하거나 실행한
것처럼 표현하지 않는다.

## 금지 사항 (전 도구 공통)

- **시크릿·고객 개인정보·미공개 재무 데이터를 프롬프트에 투입 금지** — 외부 AI 서비스 전송은 유출이다.
  연동 테스트에 실제 키가 필요하면 환경변수 참조로 우회 (값을 채팅에 붙여넣지 않는다)
- 운영 DB·운영 인프라에 대한 조작을 AI에게 위임 금지 — 사람이 직접
- 가드 우회(`--no-verify`, 권한 강제 승인)는 긴급 상황만 + 팀 리드 보고 (`pre-commit` 훅 메시지 참조)
- AI가 생성한 마이그레이션·권한 코드·금액 계산 로직은 **사람 검토 필수 지정 영역**

## 거버넌스 (도구 설정)

- 개발팀(Claude Code·Codex): repo clone → `docs/onboarding.md` §B·§D. 각 도구의
  harness-guard plugin·hook을 설치·신뢰한 상태에서 작업한다.
- 기타 도구: `AGENTS.md` + git hook + 계층 0(branch protection·CI) 범위로 제한한다.
- git-flow 작업은 수작업 대신 플러그인 커맨드 사용: `/feature-merge`, `/hotfix`, `/release`
  (게이트 절차 누락 방지 — 절차의 단일 출처는 `pr-review-gate` 스킬)
- **스킬 사용 순서**: 호출 전에 SKILL.md를 먼저 읽는다.
  `disable-model-invocation: true`가 있으면 Skill 도구로 호출하지 말고,
  SKILL.md의 bash 명령을 직접 실행한다 (모델 판단 개입 없이 절차 그대로 수행).

## AI 생성 코드의 리뷰

- 사람이 쓴 코드와 **동일한 기준** (`code-review.md`) — "AI가 짰으니 대충"도 "AI가 짰으니 의심"도 아님
- 단, AI 특유의 실패 모드를 중점 확인:
  - **과잉 구현**: 요청 안 한 추상화·옵션·방어 코드 → 삭제 요구
  - **그럴듯한 오답**: 존재하지 않는 API·옛 버전 문법 → 실행·테스트로만 검증 (눈으로 통과시키지 말 것)
  - **경계 위반**: 모듈 직접 참조, 계층 역류 → ArchUnit이 잡지만 리뷰에서도 확인
  - **테스트의 자기 충족**: 구현을 베낀 테스트(항상 통과) → 테스트가 명세를 검증하는지 확인
- PR 본문에 AI에게 준 핵심 지시를 요약 기재 (리뷰어가 의도를 알 수 있게)
